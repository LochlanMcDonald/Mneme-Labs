How it works

One decision tick, every ~2.5 seconds

The AI layer may only ever suggest — its output is data, parsed defensively. The behavior layer owns tactics. The controller layer owns physics. Here's a full pass through the loop.

Capture

Grab a camera frame (JPEG + base64). A sudden large frame-to-frame change raises a panic exception → emergency stop.

Analyze

Send the frame + task context to the vision model; get strict JSON back: description, objects, hazards, suggested actions. Calls are throttled (≥3 s apart) and cached; transient errors retry with backoff.

Sense

Read the ultrasonic sensor and build an obstacle report: distance, threshold, detected?

Decide

Second model call: given the analysis, the obstacle report, and the list of available actions — pick exactly one action, a duration, and the reasoning, as strict JSON.

Postprocess

The anti-loop watchdog and ultrasonic override get the final say. What executes can differ from what the AI asked for.

Record

Append a full JSON record of the tick to logs/decisions.jsonl — a transparent mind.

Speak

Throttled narration ("Turning left."), plus an optional AI-generated personality line.

Execute

The controller validates against an allowlist, re-checks the obstacle sensor, keeps polling it every 50 ms mid-move, and always ends in a motor stop.

Three model calls, three contracts

1 · analyze_scene()

Scene analysis

Camera frame + a mode-specific context string, prompted as a "robotics perception module." Minimum 3 s between calls; repeats inside the window reuse the cached analysis.

{ description, objects[], hazards[], suggested_actions[] }

On failure: a synthetic "AI unavailable" analysis with suggested_actions: ["stop"] — and the decision call is skipped entirely until the next tick.

2 · decide_action()

Action decision

Analysis + mode/target + the exact list of available actions + live ultrasonic data. When the sensor sees an obstacle, the prompt says so explicitly — but the veto never depends on the model listening.

{ action, duration_s, reasoning }

Hard validation: unknown action → coerced to stop; duration clamped to 0.2–3.0 s no matter what the model says. Any error → stop.

3 · generate_dialogue()

Personality dialogue

A compact context — persona, what the robot sees, hazards, the decision and what actually ran — becomes one spoken line, hard-capped at ~20 words in code.

"Corner secured. The chair leg remains undefeated."

On failure: silence. Dialogue errors never affect movement.

Safety

Six gates between the AI and the motors

Large language models are great at understanding scenes and terrible at being trusted with motors. Each layer below assumes the layers above it can be wrong.

L1Action allowlist & clamping

The controller accepts only a fixed vocabulary. A hallucinated action, a typo, malformed JSON — all coerced to stop. Durations are clamped to 0.2–3.0 seconds regardless of what the model asked for.

L2Ultrasonic veto

Before any forward motion, the sensor is read. Obstacle inside 20 cm → the command becomes stop, whatever the AI's reasoning said. The same live data is injected into the AI's prompt — but the veto doesn't depend on the model listening.

L3Mid-motion monitoring

Motion isn't fire-and-forget: during forward movement the sensor is polled every 50 ms and the move is aborted the instant an obstacle crosses the threshold. The motor stop lives in a finally block — even if the sensor read crashes mid-move, the motors stop.

L4Anti-loop watchdog

Short-term memory of recent actions and scene fingerprints detects three stuck patterns — the same action 3× in a row, A-B-A-B turn oscillation, and the scene not changing for 4 ticks. Any of them triggers a staged escape (stop → back up → turn away → forward) with temporary bans on the offending actions. A stuck forward is never answered with more forward.

L5Camera panic stop

Consecutive frames are compared in grayscale; a mean pixel change above threshold (default 35) — the signature of falling, being picked up, or a collision — raises an immediate emergency stop and a 2-second settle. A separate circuit breaker halts the robot after 5 consecutive capture failures rather than letting it run blind.

L6Fail-safe lifecycle

AI failures resolve to stop, never a guess. Ctrl-C or any unhandled exception lands in a hard-stop cleanup path that stops the motors and releases the camera — always.

"The AI proposes; deterministic code disposes." A cheap ultrasonic ping outranks the vision model. Simple counters outrank clever reasoning. Every layer is unit-tested without hardware.
Voice & personality

Meet Nimue

The robot talks. The default persona is Nimue — curious, analytical, dry-humored — and all of it is plain configuration: change the config, change the robot's character.

Under the hood: OpenAI TTS (gpt-4o-mini-tts, voice "alloy"), every utterance cached on disk so repeated phrases cost the API exactly once, played through whatever the Pi has — ffplay, mpg123, or aplay. One shared voice instance robot-wide, so the cooldowns coordinate and it never talks over itself.

Behavior modes

Four missions, one loop

All modes share the same control loop, safety systems, and watchdog — they differ in the mission briefing given to the AI, plus small tactical biases.

explore — wander with intent

"Explore the area safely. Prefer forward motion when clear." If the robot flip-flops left→right between ticks, exploration biases the next move to forward — break the dither, cover ground.

python main.py --mode explore --duration 10

detect — visual scavenger hunt

Search for anything you can name in English — the "detector" is a vision-language model, not a trained classifier. Centered and close? Stop. Not visible? Move and turn to search.

python main.py --mode detect --target "red ball" --duration 5

follow — track a target

Follow the target at a safe distance: centered and not too close → forward slowly; drifted left or right → turn to re-center; lost → rotate to reacquire.

python main.py --mode follow --target "person" --duration 5

avoid — the inverse of follow

Explore while steering away from named things. Near or centered → stop or turn away. Prefers turning over moving forward when uncertain.

python main.py --mode avoid --target "people and pets" --duration 5

There's also --mode test: a one-shot self-check that captures one frame, runs one scene analysis, and prints the result — handy for verifying camera + API key before letting the robot roam. Common flags: --duration N (minutes), --target "..." (any noun phrase), --verbose (per-tick detail including the raw vs. executed action).

Hardware

A SunFounder PiCrawler and not much else

Four articulated legs driven by hobby servos via SunFounder's Robot HAT, a Raspberry Pi on top. The software deliberately supports multiple picrawler library versions — it probes do_action, then do_step, then direct calls.

Graceful degradation is the design rule

No robot attached → MockRobot logs actions instead of moving — dry-run mode, same code, no motors
No Pi camera → OpenCV capture fallback; no camera at all → safe no-frame handling + circuit breaker
No ultrasonic sensor → obstacle checks report "unavailable"; invalid readings (≤0 or ≥400 cm) are discarded, not trusted
No audio player → speech is skipped, never crashes the loop
No API key → the program still starts; AI calls fail into safe "stop" decisions
Movement vocabulary → deliberately small: forward, backward, turn_left, turn_right, stop, plus postures (stand, sit)

Roadmap: the v4 architecture

The current robot is deliberately reactive: see, decide, act, repeat. The v4 redesign (a standalone experimental build, not yet integrated with the tested main system) explores what the same hardware can do with memory, maps, and plans.

AI planner ~5 shigh-level strategy and multi-step plans
Behavior tree ~500 msstructured action selection with explicit fallback branches
Reactive layer fastimmediate sensor-driven responses
Sensor-fusion world modelcamera, ultrasonic, and depth fused before the AI reasons — instead of a sensor veto after
Visual SLAMvisual odometry + occupancy-grid mapping: the robot builds a map and tracks its own position
Autonomous navigationA* path planning with waypoint following — "go to the kitchen" becomes a route, not a series of hunches
Monocular neural depthPyTorch depth estimation from the single RGB camera — obstacle distance across the whole field of view
Natural-language controltell the robot what to do in plain English; it grounds the instruction in what it sees
Spatial memory & learningremembering where it got stuck and which strategies failed
Watchdog execution controlleran external watchdog guaranteeing no action runs longer than 2 s even if the main loop hangs
Design principle carried overSense → Plan → Act — an honest world model from the start, deterministic safety layers underneath
