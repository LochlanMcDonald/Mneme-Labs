SEC. 02

Hardware

Four subsystems, all designed in-house. Not because we enjoy reinventing parts — because you cannot write deterministic software on top of nondeterministic hardware, and the catalog parts weren't deterministic.

HW-01 · ACTUATION

Quasi-direct-drive modules

Twenty-eight identical actuator modules in three frame sizes, each with a low-ratio planetary stage. Low gearing keeps them backdrivable: the joint feels contact through the motor windings instead of crushing through it, and torque is estimated from current — no strain gauges to drift.

  • 3 frame sizes, 1 electrical interface
  • Backdrivable · torque from current
  • Field-swappable in under 10 min
HW-02 · POWER

Hot-swap energy pack

A 2.1 kWh pack on a 96 V rail, swapped in seconds without powering down — a supercap bridge carries the machine through the exchange. The battery-management firmware is ours too, on the same audited toolchain as everything else. A robot that must dock to charge isn't working a shift.

  • 2.1 kWh · 96 V distribution
  • Zero-downtime pack exchange
  • Per-cell telemetry on the dataplane
HW-03 · COMPUTE

One board, two brains

A single carrier board holds a real-time core cluster for control and a neural accelerator for perception, sharing one memory fabric. No laptop bolted to the chest, no USB hubs, no network switch inside the torso. If a wire can be a memory bus instead, it is.

  • Real-time cores + NPU, one fabric
  • All inference on-board, no cloud
  • Single supply, single failure domain
HW-04 · SAFETY CHAIN

A stop that needs no software

The e-stop is a hardware chain: relay logic and brake actuation that fire whether or not any processor is alive. Software can request a stop; only hardware can guarantee one. Worst-case from button to mechanical hold: 0.19 seconds, measured, not simulated.

  • Relay chain independent of compute
  • Spring-applied joint brakes
  • 0.19 s to mechanical hold
SEC. 03

Software: the ICHOR runtime

In the myth, Talos ran on ichor — a single sealed vein. So does ours. ICHOR is the entire robot in one statically-linked, real-time process: one memory space, one compile-time task graph, one clock. Written from scratch, because determinism can't be bolted on afterward.

L0 · KERNEL

Verified safety kernel

A few thousand lines that own the watchdog, the brake request line, and power gating — small enough to formally verify, and it is. Every layer above L0 is allowed to crash. L0 is not. If the rest of the stack disappears mid-stride, L0 still walks the joints down to a safe hold.

PROVENFormally verified core
L1 · DATAPLANE

Shared-memory transport

Sensor frames and torque commands live in lock-free ring buffers in one address space. No serialization, no sockets, no discovery protocol, no QoS matrix — a camera frame is handed to the controller as a pointer, not a packet. Publishing a message costs nanoseconds and, critically, always costs the same nanoseconds.

0 COPIESSensor to controller
L2 · MOTION

Whole-body control at 1 kHz

A model-predictive controller fused with learned balance policies, executing on a time-triggered schedule. Every task on the 1 kHz tick has an audited worst-case execution time; the schedule is proven to fit before a release ships. Miss your budget in CI and the build fails — the robot never finds out.

<50 µSWorst-case jitter
L3 · AUTONOMY

Planning that can only suggest

Task planning, manipulation, and navigation — the only probabilistic layer in the machine. L3 proposes; L2 and L0 dispose. A plan is just a suggestion until the deterministic layers below prove it's dynamically feasible and safe. The neural network never touches a motor directly.

SANDBOXEDNo direct motor access
REPLAY

Bit-exact incident replay

Because the whole system is one deterministic process, logging its inputs is logging everything. Any anomaly from the field replays on a desk, bit-for-bit — same branches, same torques, same failure. We don't reproduce bugs. We re-run them.

1 : 1Field to desk fidelity
SEC. 04

Why there's no ROS 2 in TALOS

To be clear: ROS 2 is a remarkable way to build a research robot. It is also, in our judgment, the wrong foundation for a machine that works an unsupervised sixteen-hour shift next to people. Its flexibility is exactly the property you have to remove — a dynamic graph of nodes discovering each other over DDS middleware cannot give you a worst-case timing bound, and a worst-case timing bound is the entire product.

TALOS never ran ROS 2 and was never ported off it. ICHOR was designed first, before the first actuator spun.

Why we build

TALOS is named for the bronze giant of Greek myth — the first machine humanity ever imagined, animated by a single vein of ichor. We took the hint: one body, one vein, nothing inside it we didn't design and can't explain.

There are millions of jobs that injure the people who do them. Our position is simple: if a task is dull, heavy, or dangerous, it is a robot's job — and a robot working next to people must be deterministic all the way down. We build the whole stack because that's the only way to promise it.
